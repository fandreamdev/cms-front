# 登录 4 位验证码后端改造要求

## 接口约定

新增获取验证码接口：`GET /api/auth/captcha`，无需登录态。

统一响应的 `data` 结构如下：

```json
{
  "captchaId": "a7c2f0d8-9b70-4cde-9f1c-6ffea4ea11aa",
  "image": "data:image/png;base64,iVBORw0KGgo..."
}
```

- 验证码必须为随机生成的 **4 位数字**，允许前导零。
- `captchaId` 必须是不可预测的随机值；`image` 为可直接放入 HTML `img.src` 的 PNG/JPEG data URL。图片不得泄露答案。
- 每次请求都生成新的验证码并覆盖同一 `captchaId` 对应的旧记录（或直接签发新的 ID）。接口返回时设置 `Cache-Control: no-store, no-cache, must-revalidate`，避免浏览器或中间层缓存。

修改 `POST /api/auth/login` 的请求体：

```json
{
  "username": "admin",
  "password": "******",
  "captchaId": "a7c2f0d8-9b70-4cde-9f1c-6ffea4ea11aa",
  "captcha": "0427"
}
```

后端应校验 `captchaId` 存在、未过期且尚未使用，并以恒定时间比较验证码。无论账号、密码或验证码何项错误，都不得签发令牌。

## 存储、时效与安全

- 将验证码记录存于 Redis 等带 TTL 的服务端存储，按 `captchaId` 索引；只存答案的哈希值（可使用 HMAC），不要在日志、响应或 Cookie 中存明文答案。
- 有效期建议 5 分钟；校验尝试次数建议最多 5 次。超过次数、过期或首次校验后均立即删除记录，保证一次性使用。
- 登录校验与删除验证码必须原子执行，避免并发重放。同一 `captchaId` 即使密码错误也应作废。
- 对获取验证码及登录接口实施按 IP/用户名的限流，并记录脱敏的安全审计事件；不要记录密码、验证码明文或完整 token。
- 验证码图片应包含轻度干扰元素且使用安全随机数生成器；不要使用可预测的伪随机序列。

## 错误响应与兼容性

验证码缺失、格式错误、已过期、已使用或校验失败时，建议返回 HTTP 400，统一业务码 `CAPTCHA_INVALID`，并给出可展示的中文消息“验证码错误或已过期，请刷新后重试”。限流时返回 HTTP 429。

该字段上线后应作为登录必填项。前端在任意登录失败后会清空输入并获取一张新验证码，因此后端无需保留已提交验证码供重试。请同步更新接口文档、自动化测试和 API 网关的请求体校验规则。
